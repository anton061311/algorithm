[SECTION 06: Stack, Queue(자료구조)]
       [STACK]
        stack.pop() -> 스택 가장 위의 요소를 제거
        stack.push() -> 스택의 맨 위에 요소를 추가
        stack.peek() -> 스택 가장 위의 요소를 확인(제거X)
        stack.clear() -> 스택을 비움
        stack.isEmpty() -> true/false
        stack.size() -> 스택에 들어있는 요소의 갯수 반환
        stack.search(Object o) -> 스택의 top에서 해당 요소까지의 위치를 반환
        stack.contains(Object o) -> 스택에 해당 요소가 있는지 확인

        [QUEUE]
        queue.offer(x)
        queue.poll()
        queue.peek()
        queue.isEmpty(x)
        queue.size()
        queue.contains(x)

        그럼 코테에서 “항상 방어코드 넣는 게 좋나?”

        실무 습관으로는 “안전하게”가 좋지만, 코테에서는 보통 다음처럼 판단해.

        ✅ 방어코드 생략해도 되는 경우
        문제에서 명시적으로 보장: “올바른 괄호열”, “항상 존재”, “유효한 입력만 주어짐”
        온라인 저지 문제는 보통 이 보장이 강해서, 깔끔한 풀이가 더 선호됨(불필요 분기 감소)

        ✅ 방어코드 넣어야 하는 경우
        입력이 유효하다고 말이 없음 / 예외 케이스 가능
        스택이 비는 게 자연스러운 상태(인형뽑기, 단조 스택 문제의 초기, 데이터 스트림 등)
        peek/pop이 로직상 “언제든” 호출될 수 있음

        $ 인형뽑기는 왜 isempty()가 필요했나?

        인형뽑기에서는 스택이 비는 상황이 입력 구조상 너무 자연스럽게 생겨.
        처음 집은 인형은 스택이 비어있는데도 비교해야 할 수 있고
        어떤 move에서는 아무 인형도 못 집어서 스택 상태가 그대로일 수도 있고
        “연속 2개가 같으면 터뜨린다”라는 규칙 때문에 pop으로 비는 경우도 많음
        즉, peek()를 호출하기 전에 스택이 비었는지 체크하는 건 문제 구조상 필수야.
